# pm-prototype — 快速 Wireframe/Prototype 產生器

## Overview

這個 skill 回答「這個 idea/流程，怎麼在五分鐘內變成一個可以點著看、給人 demo 的東西」。輸入是一個功能或流程的描述（可以只是一句話，也可以是 `pm-propose` 產出的 PRD），輸出是**一份單一自包含 HTML 檔**：模擬主要使用流程、每個畫面可點擊按鈕 navigate 到下一個畫面、每個畫面標注對應的流程步驟、套用公司的 CI（顏色/字型從 `brand-tokens.css` 讀）。

重點是**流程清不清楚**，不是畫面精不精美。這不是設計稿，是拿來快速對齊「這個功能大概長怎樣、怎麼流動」的溝通工具。

**核心原則：內容脈絡（產品是什麼、流程怎麼跑）不臆測公司背景；CI 一律從 `brand-tokens.css` 讀變數，不自創顏色。**

silo skill：只做 prototype，不自動接下一步（不自動呼叫 `pm-propose`/`pm-assess`）。可以接受 `pm-propose` 產出的 PRD 當輸入，但不要求一定要有——沒有 PRD、只有一句話描述，一樣可以跑。

## Step 0 — 讀取脈絡包

在動筆之前，先讀這兩份檔案：

```
contexts/<company>/product.md
contexts/<company>/brand-tokens.css
```

- 如果不知道是哪間公司或路徑，**先問**：「這次要做 prototype 的是哪間公司？脈絡包路徑是？」不要預設任何特定公司資料夾。
- `brand-tokens.css` 缺檔尤其致命——沒有它就沒有 CI 依據，HTML 產出會變成自由發揮的通用配色，失去這個工具「套公司 CI」的意義。缺檔就停下來問要不要先補上再繼續，不要用猜測的顏色頂替。
- 如果使用者提供了 `pm-propose` 產出的 PRD（HTML 或 md 版皆可），讀進來，直接拿它的「使用者流程」段落當畫面拆解的依據，不用重問一次流程；但沒有 PRD 也完全正常，直接進 Step 1 問清楚即可。

## Step 1 — 確認輸入

如果使用者的請求或提供的 PRD 已經講清楚，直接複述你的理解讓對方確認；沒講清楚就問：

- **功能/流程名稱**：一句話描述
- **主要使用者流程**：用戶怎麼觸發、依序做什麼、每一步看到什麼（如果有 PRD，從它的「使用者流程」段落取，向對方確認拆出來的畫面清單對不對）
- **mobile 還是 desktop**：預設 mobile（375px frame），因為多數功能是 app 內流程；只有在對方明講這是「網頁版/後台/桌面工具」等明顯 web 情境時才改用 desktop 版型（更寬的 frame，例如 1024px 或不加外框直接鋪版面）
- **呈現形態：逐屏點擊還是平鋪 overview**：
  - **逐屏點擊**（預設，畫面 ≤4 時）：單一 frame，按鈕 navigate 切換畫面——適合「感受流程怎麼走」
  - **平鋪 overview**（畫面 ≥5、或對方說要拿去會議上對齊全流程時建議）：所有畫面並排橫向鋪開，屏與屏之間用箭頭＋一句話標注流程關係，每個畫面內部的按鈕仍可點（點了高亮目標畫面而非切換）——適合「一眼看懂整條流程」。跟對方確認用哪種再動手
- **資訊密度分型**：判斷這個功能屬於哪型，寫進畫面規劃——
  - **克制型**（預設）：一般流程型功能（設定、送出、確認類），每屏聚焦一件事，少容器少裝飾
  - **高密度型**：功能核心賣點是數據、AI、儀表板、追蹤類（例如收益後台、成效分析），每屏要有 ≥3 處**有內容的**差異化資訊（真實脈絡來自 `product.md` 或對方輸入——沒有真數據就標「示意值」，不憑空編一個像真的）。密度撐不起來的高密度型功能，demo 出去沒有說服力

把流程拆成畫面清單（例如：畫面 1 入口 → 畫面 2 設定 → 畫面 3 確認 → 畫面 4 完成），跟對方對一下這個拆法（含呈現形態與密度分型的判斷），再進 Step 2 產出。

## Step 2 — 產出 HTML Prototype

### 結構規範

- **單一 HTML 檔**，inline `<style>` 和 `<script>`，不外連任何資源（字型/圖示若需要用系統字型或 emoji 頂替，不外連 CDN）
- **每個畫面是一個 `<div class="screen">`**，預設只顯示第一個畫面（`display:none` 藏其他），按鈕點擊用 JavaScript 切換顯示哪個 `.screen`（簡單的 `showScreen(id)` 即可，不需要框架）
- **每個畫面標注對應的流程步驟**：畫面頂部或角落清楚寫「Step N／[這一步的名稱]」，讓看的人知道現在在流程的哪裡
- **畫面內至少有一個可點擊的元素**（按鈕/連結）navigate 到下一個畫面；最後一個畫面可以是「完成」狀態，或提供「重新開始」按鈕跳回畫面 1
- **mobile frame（預設）**：375px 寬的手機外框（圓角、模擬螢幕邊界即可，不需要畫出實體手機殼的擬真細節），畫面內容在這個 frame 裡面滾動/切換
- **desktop 版型（僅當 Step 1 判斷為 web 情境）**：不用手機外框，改用置中的桌面版型容器（例如 max-width 1024px 的卡片），畫面切換邏輯不變
- **平鋪 overview 形態（僅當 Step 1 選了 overview）**：所有 `.screen` 同時顯示、橫向並排（容器 `overflow-x: auto`），每屏上方保留 Step 標注，屏與屏之間畫箭頭＋一句話流程說明；畫面內按鈕點擊改為高亮它指向的畫面（例如加外框），讓看的人追得到流程線
- **密度落實（呼應 Step 1 的分型）**：高密度型的每一屏，檢查是否真的有 ≥3 處差異化資訊（數字、狀態、趨勢、清單——都要是這個功能真實會有的內容）；克制型則反過來檢查有沒有塞了不必要的裝飾元素。兩型都不編造看起來像真的假數據，示意值明確標注
- 風格簡潔：清楚的間距、可讀的字級、明確的按鈕，不加裝飾性動畫或花俏效果——流程清晰優先於視覺精美
- **工藝守則**：套用 `../shared/html-craft.md`（repo 內為 `skills/shared/html-craft.md`，裝機後為 `~/.claude/skills/shared/html-craft.md`）的硬底線與反 AI slop 黑名單。wireframe 定位不變（不做裝飾），但「簡潔」不等於「均一」：字重要有層次、間距要分數量級、主色當 accent 克制使用、畫面不能出現大片死白＋內容擠在角落的破版構圖

### CI 嚴格模式

- **顏色與字型只能來自 `brand-tokens.css` 定義的 CSS variables**（主色、深色、文字色、背景色、邊框色、中英文字型變數——實際變數名稱與值以該公司檔案為準，不要套用任何寫死在這份 skill 裡的色碼）。把 `brand-tokens.css` 的 `:root` 變數整段複製進這份 HTML 的 `<style>`，畫面元素一律用 `var(--xxx)` 引用，不寫死 hex。
- 若 `brand-tokens.css` 或其他脈絡檔案有明講「禁用舊色」或「衝突色」，嚴格遵守不使用，即使畫面上看起來很想用。
- 不自創任何未在 tokens 裡定義的顏色——包含看似無傷大雅的裝飾色、陰影色、漸層中間色。需要一個「稍淺/稍深」的變化時，優先用既有 token 加 CSS 的 `opacity`/`filter` 處理，而不是手動配一個新 hex。

### 存檔

存至 `contexts/<company>/outputs/<日期>-<功能slug>-prototype.html`。

## Step 3 — 顏色自審（產出後必做）

產出 HTML 後，列出檔案裡出現的**每一個 `#hex` 值**，逐一比對是否都能在 `brand-tokens.css` 的 `:root` 變數裡找到來源（直接是某個變數的值，或是該變數配 `opacity`/`filter` 後的呈現，而不是另外編出來的新色碼）。

任何找不到來源的顏色，一律刪除或改用既有 token 表達——不接受任何新配色，即使它只是裝飾性的。同時確認畫面裡沒有出現任何被明講禁用的舊色。

自審結果簡短列出來（例如「本次用到 N 組 hex，逐一列出每個 hex 值對應到 `brand-tokens.css` 的哪個變數」），讓對方一眼看到這份 prototype 確實只用了公司自己的顏色。

## Step 4 — 產出驗證（交付前必做，不能只靠讀原始碼）

HTML 寫完 ≠ 做完。交付前必須**實際渲染並看過**這份產出：

1. 跑驗證 script（在本 skill 目錄的上一層 `shared/scripts/verify.py`；repo 內為 `skills/shared/scripts/verify.py`，裝機後為 `~/.claude/skills/shared/scripts/verify.py`）：
   ```bash
   python3 <shared>/scripts/verify.py "<輸出檔路徑>"
   ```
   它會用 Playwright 打開檔案、截圖（存到輸出檔旁的 `screenshots/`）、抓 console/page error。
2. **Page errors 必須為 0**——有 JS 錯誤就修掉重跑，不帶著錯誤交付。
3. **用 Read 工具打開截圖，肉眼檢查**：畫面沒破版、CI 顏色如預期、每個畫面的 Step 標注清楚可讀、按鈕看起來可點。發現問題就修，修完重跑驗證。
4. 驗證通過後，在交付訊息中附一句驗證結果（例如「已渲染驗證：無 JS 錯誤，截圖已確認版面正常」）。

**降級**：若環境沒有 playwright（script 會明確報錯），不要假裝驗證過——明講「本次未經渲染驗證」，並請對方用瀏覽器開啟確認。點擊 navigate 流程 script 測不到，交付時仍提醒對方實際點一遍。

## Step 5 — 詢問後續

Prototype 完成後問對方：「Prototype 完成了，[N] 個畫面可以點著看。接下來要做什麼？
1. 調整某個畫面的流程或內容
2. 先停在這裡，我要先拿去給人看」

不自動接續其他 skill（例如不主動建議「要不要跑 pm-propose 產 PRD」），等對方明確要求再說。
