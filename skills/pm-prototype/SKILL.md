# pm-prototype — Demo 級 Prototype 產生器

## Overview

這個 skill 回答「這個 idea/流程，怎麼快速變成一個能拿去 demo 給人看的東西」。輸入是一個功能或流程的描述（可以只是一句話，也可以是 `pm-propose` 產出的 PRD），輸出是**一份單一自包含 HTML 檔**，有兩種模式：

- **demo 級（預設）**：平鋪 4-6 個主畫面、每屏都能真實交互（切畫面/彈 modal/開關 toggle）、精確手機外框、套公司 CI——能直接拿去給 CEO/stakeholder demo 的質感。**Prototype 是 demo 現場，靜態擺拍沒有說服力。**
- **wireframe 快速模式**：對方明講「只要對流程」「線框就好」「越快越好」時走——單 frame 逐屏點擊、簡化外框、灰塊佔位，流程清晰優先於視覺質感。

**核心原則：內容脈絡（產品是什麼、流程怎麼跑）不臆測公司背景；CI 一律從 `brand-tokens.css` 讀變數，不自創顏色。**

silo skill：只做 prototype，不自動接下一步（不自動呼叫 `pm-propose`/`pm-assess`）。可以接受 `pm-propose` 產出的 PRD 當輸入，但不要求一定要有——沒有 PRD、只有一句話描述，一樣可以跑。

## Step 0 — 讀取脈絡包與守則

在動筆之前，先讀：

```
contexts/<company>/product.md
contexts/<company>/brand-tokens.css
```

- 如果不知道是哪間公司或路徑，**先問**：「這次要做 prototype 的是哪間公司？脈絡包路徑是？」不要預設任何特定公司資料夾。
- `brand-tokens.css` 缺檔尤其致命——沒有它就沒有 CI 依據，HTML 產出會變成自由發揮的通用配色，失去這個工具「套公司 CI」的意義。缺檔就停下來問要不要先補上再繼續，不要用猜測的顏色頂替。
- 如果使用者提供了 `pm-propose` 產出的 PRD（HTML 或 md 版皆可），讀進來，直接拿它的「使用者流程」段落當畫面拆解的依據，不用重問一次流程；但沒有 PRD 也完全正常，直接進 Step 1 問清楚即可。

共用守則（`../shared/`；repo 內為 `skills/shared/`，裝機後為 `~/.claude/skills/shared/`）：

- `html-craft.md`——所有 HTML 的排版工藝底線（兩種模式都要遵守）
- `prototype-craft.md`——demo 級模式的專屬守則（平鋪＋狀態機、設備框、真圖、點擊測試），**demo 級動筆前必讀**

## Step 1 — 確認輸入與模式

如果使用者的請求或提供的 PRD 已經講清楚，直接複述你的理解讓對方確認；沒講清楚就問：

- **功能/流程名稱**：一句話描述
- **主要使用者流程**：用戶怎麼觸發、依序做什麼、每一步看到什麼（如果有 PRD，從它的「使用者流程」段落取，向對方確認拆出來的畫面清單對不對）
- **模式判斷**：預設 **demo 級**。只有對方明講「只要對流程」「線框/wireframe 就好」「越快越好、不用漂亮」時才走 wireframe 快速模式；拿不準就用一句話確認（「這次要能拿去 demo 的質感，還是快速線框對流程就好？」）
- **mobile 還是 desktop**：預設 mobile；只有在對方明講這是「網頁版/後台/桌面工具」等明顯 web 情境時才改用 desktop 版型（更寬的容器，例如 1024px，不加手機外框）
- **資訊密度分型**：判斷這個功能屬於哪型，寫進畫面規劃——
  - **克制型**（預設）：一般流程型功能（設定、送出、確認類），每屏聚焦一件事，少容器少裝飾
  - **高密度型**：功能核心賣點是數據、AI、儀表板、追蹤類（例如收益後台、成效分析），每屏要有 ≥3 處**有內容的**差異化資訊（真實脈絡來自 `product.md` 或對方輸入——沒有真數據就標「示意值」，不憑空編一個像真的）。密度撐不起來的高密度型功能，demo 出去沒有說服力
- **視覺方向模糊時**：對方說「不知道要長怎樣」「給我幾個方向看看」，或你發現同一份需求有多種合理的版面走法且對方沒表態——走 `../shared/variations.md` 的**設計方向顧問模式**（出三版真實視覺讓對方選，不丟文字選擇題）。方向清楚就直接往下，不要沒事跑三版。

把流程拆成畫面清單（例如：畫面 1 入口 → 畫面 2 設定 → 畫面 3 確認 → 畫面 4 完成），跟對方對一下這個拆法（含模式與密度分型的判斷），再進 Step 2 產出。

## Step 2 — 產出 HTML Prototype

### 兩種模式共用的結構規範

- **單一 HTML 檔**，inline `<style>` 和 `<script>`，**不外連任何資源**（字型用系統字型或 tokens 定義；圖片一律 base64 內嵌；不外連 CDN——包含 React/Babel，那是 pm-demo 動畫中間產物的專屬例外，prototype 是交付物，不適用）
- **每個畫面標注流程步驟**：畫面上方或角落清楚寫「Step N／[這一步的名稱]」，讓看的人知道現在在流程的哪裡
- **desktop 版型（僅當 Step 1 判斷為 web 情境）**：不用手機外框，改用置中的桌面版型容器（例如 max-width 1024px 的卡片）
- **密度落實（呼應 Step 1 的分型）**：高密度型的每一屏，檢查是否真的有 ≥3 處差異化資訊（數字、狀態、趨勢、清單——都要是這個功能真實會有的內容）；克制型則反過來檢查有沒有塞了不必要的裝飾元素。兩型都不編造看起來像真的假數據，示意值明確標注
- **工藝守則**：套用 `../shared/html-craft.md` 的硬底線、字重/空間/accent 紀律、反 AI slop 黑名單、品位錨點

### demo 級模式（預設）——依 `../shared/prototype-craft.md` 執行

- **交付形態**：平鋪 4-6 個主畫面＋每台可交互，不問二選一（特例見 prototype-craft §1）
- **手機外框**：複製 `../shared/assets/phone-frame.html` 的 `.pf-*` 樣式與骨架（iPhone 15 Pro 精確規格），**禁止手寫** Dynamic Island／status bar／home indicator
- **每台是獨立迷你狀態機**（vanilla JS 骨架見 prototype-craft §2）：按鈕可點、畫面可切、modal 能彈、開關能 toggle，可點元素有 hover/active 回饋
- **真圖誠實性測試**（prototype-craft §4）：內容圖（去掉資訊有損）才放、base64 內嵌；裝飾圖不放；拿不到真圖用誠實灰塊佔位
- **品位錨點**：動筆前按 html-craft「品位錨點」節定調；120% 細節簽名選在 demo 的高潮畫面

### wireframe 快速模式（對方明講才走）

- **單一 frame 逐屏點擊**：每個畫面是一個 `<div class="screen">`，預設只顯示第一個畫面，按鈕點擊用簡單的 `showScreen(id)` 切換；最後一屏是「完成」狀態或提供「重新開始」
- 畫面 ≥5 或要拿去會議對齊全流程時，可改平鋪 overview（所有畫面並排、屏間畫箭頭＋一句話流程說明、按鈕點擊高亮目標畫面）——跟對方確認用哪種
- **簡化外框**：375px 寬的簡單圓角手機框即可，不需要精確設備細節
- 圖片一律灰塊＋文字標籤誠實佔位；風格簡潔，不加裝飾動畫——但「簡潔」不等於「均一」：字重要有層次、間距要分數量級、主色當 accent 克制使用、畫面不能出現大片死白＋內容擠角落的破版構圖

### CI 嚴格模式（兩種模式都要）

- **顏色與字型只能來自 `brand-tokens.css` 定義的 CSS variables**（主色、深色、文字色、背景色、邊框色、中英文字型變數——實際變數名稱與值以該公司檔案為準，不要套用任何寫死在這份 skill 裡的色碼）。把 `brand-tokens.css` 的 `:root` 變數整段複製進這份 HTML 的 `<style>`，畫面元素一律用 `var(--xxx)` 引用，不寫死 hex。
- 唯一例外：`phone-frame.html` 設備外殼本身的顏色（bezel 黑、status bar 元件）——那是「手機硬體」不是畫面內容，維持原樣不要套品牌色。
- 若 `brand-tokens.css` 或其他脈絡檔案有明講「禁用舊色」或「衝突色」，嚴格遵守不使用，即使畫面上看起來很想用。
- 不自創任何未在 tokens 裡定義的顏色——包含看似無傷大雅的裝飾色、陰影色、漸層中間色。需要一個「稍淺/稍深」的變化時，優先用既有 token 加 CSS 的 `opacity`／`color-mix()` 處理，而不是手動配一個新 hex。

### 存檔

存至 `contexts/<company>/outputs/<日期>-<功能slug>-prototype.html`。

## Step 3 — 顏色自審（產出後必做）

產出 HTML 後，列出檔案裡出現的**每一個 `#hex` 值**，逐一比對是否都能在 `brand-tokens.css` 的 `:root` 變數裡找到來源（直接是某個變數的值，或是該變數配 `opacity`／`filter`／`color-mix()` 後的呈現，而不是另外編出來的新色碼）。設備外殼的 `.pf-*` 顏色屬於前述例外，列出時標注即可。

任何找不到來源的顏色，一律刪除或改用既有 token 表達——不接受任何新配色，即使它只是裝飾性的。同時確認畫面裡沒有出現任何被明講禁用的舊色。

自審結果簡短列出來（例如「本次用到 N 組 hex，逐一列出每個 hex 值對應到 `brand-tokens.css` 的哪個變數」），讓對方一眼看到這份 prototype 確實只用了公司自己的顏色。

## Step 4 — 產出驗證（交付前必做，不能只靠讀原始碼）

HTML 寫完 ≠ 做完。交付前必須**實際渲染並看過**這份產出：

1. 跑驗證 script（`../shared/scripts/verify.py`）：
   ```bash
   python3 <shared>/scripts/verify.py "<輸出檔路徑>"
   ```
   它會用 Playwright 打開檔案、截圖（存到輸出檔旁的 `screenshots/`）、抓 console/page error。
2. **Page errors 必須為 0**——有 JS 錯誤就修掉重跑，不帶著錯誤交付。
3. **demo 級加跑 3 項最小點擊測試**（prototype-craft §6）：挑最關鍵的三個交互（navigate／modal／toggle）用 playwright 點過，點擊後 pageerror = 0、截圖確認狀態真的變了。
4. **用 Read 工具打開截圖，肉眼檢查**：畫面沒破版、CI 顏色如預期、每個畫面的 Step 標注清楚可讀、按鈕看起來可點；對著截圖回答 html-craft 第 10 節的四題 craft 自檢。發現問題就修，修完重跑驗證。
5. 驗證通過後，在交付訊息中附一句驗證結果（例如「已渲染驗證：無 JS 錯誤，3 項點擊測試通過，截圖已確認版面正常」）。

**降級**：若環境沒有 playwright（script 會明確報錯），不要假裝驗證過——明講「本次未經渲染驗證」，並請對方用瀏覽器開啟確認、實際點一遍交互。

## Step 5 — 詢問後續

Prototype 完成後問對方：「Prototype 完成了，[N] 個畫面可以點著看。接下來要做什麼？
1. 調整某個畫面的流程或內容
2. 出 2-3 個不同視覺方向的版本比較（走 `../shared/variations.md`）
3. 對這份產出做一次專家評審（走 `../shared/critique.md`）
4. 先停在這裡，我要先拿去給人看」

不自動接續其他 skill（例如不主動建議「要不要跑 pm-propose 產 PRD」），等對方明確要求再說。
