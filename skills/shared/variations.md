# variations.md — 設計方向顧問模式與變體機制

`pm-prototype` 與 `pm-propose`（report）共用的多版本機制。一套流程、兩個觸發點：

| 觸發 | 何時 | 名稱 |
|---|---|---|
| **顧問模式**（自動） | 對方說「不知道要長怎樣」「給我幾個方向」，或同一份需求有多種合理版面走法、對方沒表態 | 設計方向顧問 |
| **變體**（明講才走） | 對方明確要求「出幾版讓我選」「想比較不同做法」 | 設計變體 |

**預設不開**——方向清楚就直接單版做到好，這個機制約三倍成本（時間與 token），別沒事就跑。但一旦觸發就走完整流程，不許只做一版交差。

> 方法論蒸餾自 [huashu-design](https://github.com/alchaincyf/huashu-design)(alchaincyf，MIT License）的 Fallback 顧問模式與 variations 哲學，依 pm-almighty-box 的 CI 鎖定前提改寫。

## 0. 選擇無效鐵律（本機制的存在理由）

**絕不讓對方在「只有文字描述、沒看到視覺」時選方向**——他沒有依據，選了也是猜。所以：

- ❌ 不丟文字選擇題（「你要 A 極簡風還是 B 資訊密集風？」）
- ✅ 直接做出 **3 個真實版本**，截圖並排擺出來，讓對方選「看得見的東西」

推論：觸發顧問模式後不要停下來繼續追問風格偏好——對方答不出來才觸發的。把問題變成看得見的三版，就是答案。

## 1. 跟原專案的關鍵差異：CI 鎖死，變的是結構不是顏色

原專案的三版可以換整套配色字體；我們不行——**顏色與字型永遠只來自 `brand-tokens.css`**（CI 嚴格模式，三版都要過顏色自審）。三版的差異化軸是：

- **布局骨架**（結構性差異，硬要求：三版至少一項互異——導航方式／構圖重心／內容區結構，不許兩版共用同一骨架只微調間距）
- **排印層次**（字級對比拉多開、display 大而輕還是重而小、密度節奏疏密怎麼分布）
- **視覺母題**（從這次內容長出來的視覺元素——每版找一個不同的母題詮釋）
- **120% 細節簽名的位置**（每版選不同的地方精緻）

## 2. 流程

### 2.1 固化 spec（三版的唯一共同輸入）

把已澄清的需求寫成一份 **≥300 字**的 spec，寫薄了三版都會飄。必須覆蓋：功能/產品是什麼、受眾與使用場景、畫面清單（prototype）或內容大綱（report）、密度分型、`brand-tokens.css` 路徑、輸出尺寸/形態（三版必須統一，否則無法橫向對比）、已知禁忌（禁用舊色等）、每版的母題假設方向。

### 2.2 三套邏輯並行，各出一版真實視覺

用 3 個並行 subagent（各自獨立 context、只讀 spec、互不參考——避免趨同）；不支援 subagent 的環境改串行，每版開跑前只讀 spec、不看已產出的版本。三套邏輯：

**邏輯一・結構輪盤（隨機錨）**：跑 `date +%S` 取秒數 `% 8 + 1`，從下表取對應骨架執行。作用：用外部隨機源打破模型「每次都偷選安全置中卡片流」的確定性偏好。

| # | 骨架原型 | 適合 |
|---|---|---|
| 1 | 分欄對照（左右兩案並排比） | 對比、前後差異 |
| 2 | 縱走時間軸／步驟導軌 | 流程、時程 |
| 3 | 卡片牆（等權重網格） | 多項並列、清單 |
| 4 | 全幅焦點＋側欄註解 | 一個主畫面/主數字撐場 |
| 5 | 表格主導（數據為版面重心） | 排序、規格、指標 |
| 6 | 精靈式分步（一屏一事，進度導航） | 設定流、onboarding |
| 7 | 儀表格（KPI strip＋圖表＋明細） | 後台、成效 |
| 8 | 敘事長卷（章節式縱向敘事） | 提案說服、報告 |

抽到明顯不合的骨架（例如「修改暱稱」抽到儀表格）就重抽一次，最多重抽一次，第二次抽到什麼就想辦法詮釋——勉強的詮釋常常就是新意的來源。

**邏輯二・現實參照（標杆錨）**：選一個你明確知道「這類流程做得極好」的真實產品（付款流想 Stripe checkout、訂閱設定想 App Store 訂閱頁的層次……），能 WebSearch 就先驗證你對它設計語言的記憶，拆解它的**結構與層次**（不是顏色），遷移到這次內容上。交付標注參照了誰、遷移了什麼。

**邏輯三・設計哲學（品味錨）**：認真想——假如預算沒有上限，最適合為「這間公司、這個功能」操刀的設計師/工作室是誰？（按產品調性選：原研哉的留白、Stripe 團隊的層次工程、IDEO 的流程敘事……）用那套哲學從頭思考版面。交付標注用了誰的哲學、體現在哪。

三版共同規範：

- 用**真實內容**（同一份 spec 的內容，非 Lorem），只換設計邏輯，方便橫向對比
- 各自完整遵守該 skill 的產出規範（prototype 走 prototype-craft、report 走 html-craft）＋ CI 顏色自審
- 可讀性硬底線不豁免（html-craft §1）——「安靜留白」做過頭變成大片死白＋微縮字，第一眼像渲染壞了
- 檔名：`<日期>-<功能slug>-<產出型>-vA/vB/vC.html`，同資料夾

### 2.3 截圖並排讓對方選

三版都跑 `verify.py`（page errors = 0）取得截圖，**一次貼出三張截圖**，每版三行說明：用了哪套邏輯（抽到幾號骨架/參照了誰/用了誰的哲學）、母題是什麼、一句話為什麼這樣走。然後問對方：

「三版都是真的可以開的檔案。要：1. 選一版深化 2. 混合（例如 vA 的骨架＋vC 的密度） 3. 微調某版 4. 都不對，我重跑三版」

### 2.4 收斂

對方選定（或指定混合）後，以該版為底收斂成正式版：檔名去掉 `-vX` 後綴、走該 skill 完整的驗證流程（顏色自審＋渲染驗證＋點擊測試）、其餘兩版檔案保留不刪(對方可能回頭要）。收斂版才是交付物，三個 v 版是過程產物。
